### [001] 2026-07-01 · 保存后用户看不到历史记录

- **状态**：已修复
- **类型**：产品体验 · 信息反馈缺失
- **描述**：用户保存事件记录后返回首页，看不到自己刚才记了什么。用户会产生“记录去哪了？”的困惑，降低对工具的信任。
- **归因**：原型第一版缺少“记录归档与结果反馈”机制，用户完成核心操作后没有获得任何确认可见的结果呈现。
- **修复方案**：增加左侧侧边栏，按“记仇”和“不舒服的互动”分类归档，每条记录以日期+对象+一句话简述的形式呈现。
- **修复后效果**：用户保存后可立即在侧边栏看到记录简述，确认“它被妥善保管了”。

### [002] 2026-07-01 · 缺少分析功能的预期引导

- **状态**：已修复
- **类型**：产品体验 · 功能连续性缺失
- **描述**：用户完成多条记录后，不知道积累这些记录“接下来能做什么”。工具变成纯记录本，没有兑现“帮你看见脉络”的承诺。
- **归因**：缺少阶段性提示和分析入口的预期引导。
- **修复方案**：在侧边栏增加累计条数提示（3条时温和提醒，5条时出现“查看脉络分析”按钮），让用户知道记录有目的。
- **修复后效果**：用户有清晰的“下一步”预期，记录的动机得到强化。

### [003] 2026-07-01 · 表单字段措辞不够引导性

- **状态**：已修复
- **类型**：产品体验 · 文案与引导
- **描述**：“对方做了什么”太笼统；“我不满的地方”让用户无从下手；“事件的结局”指向不清。
- **归因**：字段设计偏抽象，缺少对用户回忆状态的共情式引导。
- **修复方案**：拆分“对方做了什么”为行为+语言两字段；将“我不满的地方”替换为3个引导性问题（下意识想法/具体感受/联想画面）；细化“事件的结局”为“当时你或对方做了什么/说了什么，让这次互动结束了？”。
- **修复后效果**：用户能更顺畅地进入回忆和记录状态。

### [004] 2026-07-01 · 表单字段逻辑与功能联动全面优化

- **状态**：已修复
- **类型**：产品体验 · 表单设计与引导系统
- **描述**：
  1. 表单字段存在重复和逻辑不顺的问题，未按心理复盘的自然流程排序；
  2. 联想模块未与历史记录联动，用户无法快速参考过去与同一对象的事件；
  3. 结束引导语单一，未体现持续记录的价值，缺少随机化机制；
  4. 初始两个选项“我要记仇”和“我有一次不舒服的互动”界定模糊。
- **影响**：用户填写时感到重复或困惑，完成后缺乏继续使用动力，初始选择时犹豫。
- **归因**：表单设计偏重事件记录，忽视了心理复盘的9步自然链条；引导语缺少语料库机制；选项描述未区分用户画像和核心需求。
- **修复方案**：
  1. 表单重组为9字段心理复盘链：对象→事件开始→对方言行→情绪起点→自动思维→我的反应与对方反应→结束方式→联想→评分
  2. 联想模块增加“查看与[对象]相关的历史记录”按钮
  3. 结束引导改为5条随机语料库（结构：鼓励+持续记录的好处）
  4. 初始选项重新界定：“记仇”=有明确对象，“不舒服的瞬间”=不确定是什么问题
- **修复后效果**：表单逻辑与用户自然复盘流程一致；联想模块与历史数据打通；引导语多样化且每次都有“下一步”的动机强化；选项清晰，用户可快速选择。

### [005] 2026-07-01 · 侧边栏布局失衡与归档信息展示不完整

- **状态**：已修复
- **类型**：产品体验 · 页面布局与信息展示
- **描述**：
  1. 归档侧边栏位于页面左侧，导致整体页面结构失衡，视觉重心偏移，不符合主流网页的阅读习惯；
  2. 归档栏内的事件记录显示不完整，存在折叠或截断问题，用户无法一眼看到"对象+事件简介"的完整信息。
- **影响**：用户浏览体验不舒适，归档信息的可读性和可用性降低，削弱了"记录被妥善保管"的安全感。
- **归因**：初版布局未考虑视觉平衡和主流网页设计规范；归档信息展示未针对侧边栏宽度做适配优化。
- **修复方案**：
  1. 将侧边栏从左侧移至右侧，主内容区占70-75%，归档栏占25-30%，符合主流网页"主内容+侧边栏"的布局习惯；
  2. 归档栏内记录显示格式统一为"对象 + 事件简介（前15-20字+…）"，确保在栏内完整显示，不折叠、不隐藏。
- **修复后效果**：页面布局视觉平衡，阅读体验更自然；归档信息一目了然，用户能快速定位历史记录。

### [006] 2026-07-01 · 布局微调后视觉重心与顶部对齐问题

- **状态**：已修复
- **类型**：产品体验 · 页面布局微调
- **描述**：
  1. 归档栏右移并缩小后，主体内容区偏左，页面视觉重心失衡；
  2. 主体内容区微调至中央后，整体感觉仍稍微偏左偏上，需要向右和向下移动"约5个中文字的距离"；
  3. 主体内容区下移后，右侧归档栏顶部未同步下移，导致左右两侧顶部不在同一水平线上。
- **影响**：页面视觉不够协调，用户视线落点不够舒适，影响整体产品质感。
- **归因**：多次布局调整后，各模块之间的相对位置未做统一对齐校准。
- **修复方案**：
  1. 主体内容区向右微调，增加左侧留白；
  2. 主体内容区向下微调，增加顶部留白；
  3. 右侧归档栏顶部跟随主体内容区同步下移，保持左右两侧顶部齐平。
- **修复后效果**：页面整体视觉平衡，主体内容位于视觉焦点，左右两侧顶部完全对齐，观感舒适自然。

### [007] 2026-07-01 · 归档查看与编辑交互重构

- **状态**：已修复
- **类型**：产品体验 · 归档查看与编辑交互重构
- **描述**：
  1. 归档记录的查看页面与编辑表单使用同一套排版，未区分"浏览"和"修改"两种用户场景；
  2. 查看页面缺少"返回主页"按钮，用户只是想快速确认记录是否保存成功，却需要手动导航回去；
  3. 查看页面信息逐行排列，空间分布过大，无法让用户一眼抓住重点；
  4. 事件摘要由系统自动截取字段生成，在没有AI提炼的情况下，显示效果不自然；
  5. Dashboard顶部缺少清晰的事件标识，用户进入后无法第一时间确认"这是哪次记录"。
- **影响**：用户回顾历史记录时体验碎片化，信息获取效率低，缺少"快速浏览后离开"的轻量路径。
- **归因**：
  1. 查看与编辑未做场景分离设计；
  2. 信息展示未采用Dashboard式的高密度布局；
  3. 用户路径缺少"返回主页"的闭环；
  4. 标题生成依赖字段截取，缺少用户自定义入口。
- **修复方案**：
  1. 查看页面重构为Dashboard式布局：顶部评分与摘要、左列事实区、右列感受区、底部连接区；
  2. Dashboard顶部增加"编辑"和"返回主页"两个按钮，支持快速离开；
  3. 新增"给这次记录起个名字"选填字段，用户可自定义标题；
  4. 侧边栏和Dashboard顶部优先展示用户自定义标题，未填写时用默认格式兜底；
  5. 编辑表单保持原有宽松排版不变，与Dashboard紧凑布局形成场景区分。
- **修复后效果**：查看与编辑场景分离，Dashboard一眼可见评分、事实、感受三大块；用户可快速浏览后返回主页；自定义标题让每条记录更有个人的温度。

### [008] 2026-07-01 · 梦境模块新增与首页入口调整

- **状态**：已修复
- **类型**：产品功能扩展 · 新模块新增与入口调整
- **描述**：
  1. 产品仅有事件记录功能，缺少对用户其他认知探索路径的支持。用户在疗愈过程中除了记录事实，还会记录梦境，产品需要体现"支持网络"的理念；
  2. 首页新增梦境入口后，信息过多，用户面对多个卡片时选择负担加重；
  3. 首页按钮文字"开始记录"表意不够精准，未与"记录梦境"形成清晰的功能区分。
- **影响**：产品功能单一，无法承载用户多元的自我探索需求；首页入口复杂化，削弱了"快速开始"的体验。
- **归因**：
  1. 初版设计聚焦于事件记录，未将梦境纳入"认知支持网络"的产品架构；
  2. 入口设计从卡片式改为按钮式时，未及时调整文字和位置。
- **修复方案**：
  1. 新增完整的"梦境记录"模块，包含：梦境记录表单（10个字段，覆盖基本信息、梦境内容、情感、醒来感受、现实背景）、侧边栏梦境归档文件夹、梦境Dashboard预览页面、事件Dashboard底部的梦境关联；
  2. 首页入口从多卡片式改为两个简洁按钮："记录事实"和"记录梦境"，按钮放置在所有说明文字下方；
  3. 梦境模块的设计原则：不做过重的理性分析，聚焦于"记住印象深刻的片段"和"醒来后的感受"；
  4. 事件与梦境通过Dashboard底部关联模块实现"支持网络"的互相链接。
- **修复后效果**：首页简洁清晰，两个入口功能区分明确；梦境模块轻量化、感受导向，与事件记录形成互补；产品从单点工具扩展为"白天记录事实+夜晚记录梦境"的双轨认知支持网络。

### [009] 2026-07-01 · 编辑状态保护与断点续填

- **状态**：已修复
- **类型**：产品功能新增 · 编辑状态保护与断点续填
- **描述**：
  1. 用户在填写事件记录或梦境记录表单时，如果中途离开（关闭页面、返回首页、误触退出），所有已填写的内容会丢失；
  2. 用户无法在归档栏中看到"未完成"的记录，也无法从上次中断的地方继续填写；
  3. 对于NPD创伤场景的用户，写了一半的内容消失可能触发被忽视、被抹去的创伤感受，降低对工具的信任。
- **影响**：用户填写表单时缺乏安全感，可能因为担心内容丢失而不敢开始记录，或在意外退出后产生挫败感，降低产品留存率。
- **归因**：
  1. 初版只有"保存"按钮一种提交方式，缺少中间状态的保护机制；
  2. 未区分"正式记录"和"草稿"两种数据状态；
  3. 用户返回页面时，缺少对未完成草稿的识别和恢复引导。
- **修复方案**：
  1. 新增表单自动保存功能：用户停止输入1-2秒后，自动将当前已填内容保存为草稿（localStorage单独存储）；
  2. 侧边栏归档文件夹中，新增"未完成"状态标签（柔和橙色小标签），显示日期+对象/标题+"（未完成）"；
  3. 首页新增"你有一条未完成的记录，要继续吗？"提示，点击"继续填写"可恢复到退出前的表单状态；
  4. 用户正式保存后，对应草稿自动清除；返回主页时草稿保留；无输入不产生草稿；
  5. 事件记录草稿和梦境记录草稿分开存储，互不影响。
- **修复后效果**：用户可放心填写，中途离开内容不丢失；侧边栏清晰标记未完成记录；首页温和引导用户继续填写；产品安全感显著提升。

### [010] 2026-07-01 · 草稿系统重构与导航逻辑完善

- **状态**：已修复
- **类型**：产品功能修复与优化 · 草稿系统重构与导航逻辑完善
- **描述**：
  1. 草稿存储只能保存1条，新草稿会覆盖旧草稿，导致用户无法同时保留多条未完成记录（比如同时有一条未完成的事件记录和一条未完成的梦境记录）；
  2. 用户点击"保存"后，草稿未自动清除，导致侧边栏仍显示"未完成"标签，再次点击保存会生成重复记录；
  3. 侧边栏未完成标记在文字后附加"（未完成）"字样，与前方标签重复，视觉冗余；
  4. 首页未完成提示只支持1条草稿的场景，无法告知用户有多条未完成记录；
  5. 编辑页面缺少"返回"按钮，用户在填写过程中如果想返回上一级，只能通过浏览器后退或直接回首页；
  6. 事件记录表单的返回逻辑直接跳回首页，用户如果在填写中发现选错了记录类型（比如选了"不舒服的瞬间"但写着写着觉得应该归为"记仇"），无法返回类型选择页重新选择。
- **影响**：
  1. 用户无法安全地同时保留多条未完成记录，草稿丢失风险高；
  2. 重复记录破坏数据准确性，用户对归档系统的信任下降；
  3. 页面导航路径不完整，用户在填写过程中缺少灵活调整的空间。
- **归因**：
  1. 草稿存储设计为单条覆盖模式，未考虑用户同时进行多种记录的场景；
  2. 草稿清除逻辑与页面跳转逻辑未完全解耦，导致特定路径下清除失效；
  3. 侧边栏标记样式经历多次调整，文字标签出现重复；
  4. 编辑页面初始设计未包含返回导航，用户路径缺少闭环。
- **修复方案**：
  1. 草稿存储改为数组模式，支持多条草稿并存，每条草稿有唯一ID、类型、时间戳；
  2. 保存时只清除当前草稿，其他草稿不受影响；
  3. 首页未完成提示升级为动态显示（"你有X条未完成的记录"），支持多条草稿场景，点击默认打开最新修改的草稿，并提供"查看全部未完成记录"入口；
  4. 侧边栏未完成标记保留圆角边框浅橙色"未完成"标签样式，删除后面文字中的"（未完成）"重复描述；
  5. 所有编辑页面标题下方新增"← 返回"按钮；
  6. 事件记录表单的返回逻辑指向类型选择页（"记仇"/"不舒服的瞬间"），允许用户重新选择记录类型；梦境记录表单的返回逻辑指向首页。
- **修复后效果**：
  1. 用户可同时保留多条未完成记录，草稿数据安全不丢失；
  2. 保存后草稿准确清除，不再产生重复记录；
  3. 侧边栏标记简洁清晰，标签与文字各司其职；
  4. 首页能准确告知用户未完成记录数量，引导用户继续编辑；
  5. 编辑页面导航路径完整，用户可灵活返回上一级，体验闭环。

### [011] 2026-07-01 · 记录类型灵活修改与视觉标识

- **状态**：已修复
- **类型**：产品功能新增 · 记录类型灵活修改与视觉标识
- **描述**：
  1. 用户在事件记录表单中，无法在填写过程中修改记录类型。如果用户最初选了"不舒服的瞬间"，但写着写着发现应该归类为"记仇"，必须返回类型选择页重新选择，操作路径断裂；
  2. 已归档记录在Dashboard预览页面中，标题旁边没有类型标识，用户无法一眼判断这条记录属于哪个分类；
  3. 类型标签在编辑页面已存在，但仅作为静态展示，不具备交互功能。
- **影响**：
  1. 用户在填写过程中如果发现类型选错，需要中断当前记录、返回上一级、重新进入，体验割裂。NPD受害者在记录时情绪可能波动，中断操作会降低记录意愿；
  2. 回顾历史记录时，缺少类型标识让用户需要额外记忆或点进去才能确认分类，信息获取效率低。
- **归因**：
  1. 初版设计将记录类型视为"入口时确定"的固定属性，未考虑用户在记录过程中对事件性质的认知可能发生变化；
